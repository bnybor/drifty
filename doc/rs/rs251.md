# drifty — `rs251` block codec

A **Reed–Solomon** code over GF(251), presented through the block codec
interfaces ([`block_encoder.h`](../../include/drifty/block_encoder.h),
[`block_decoder.h`](../../include/drifty/block_decoder.h)). It is a systematic
RS(n, k): k information symbols become an n-symbol codeword with n − k parity
symbols, and the decoder corrects errors and erasures while
`2·errors + erasures ≤ n − k`. As a hard-decision **outer** block code it pairs
naturally with the convolutional **inner** codecs in [`cc/`](../cc/README.md).

For the symbol alphabet and what each interface means, see
[Data-flow semantics](../data_flow_semantics.md).

## Algorithm

A symbol is an element of GF(251) — a value in `0..250`. The message is treated
as the evaluations of a degree-`< k` polynomial; the codeword is that polynomial
sampled at n points, so the first k codeword symbols are the message (systematic)
and the remaining n − k are parity. The decoder reconstructs the polynomial from
the received symbols, correcting any mix of **errors** (wrong symbols at unknown
positions, each costing 2 check symbols) and **erasures** (symbols flagged
unknown, each costing 1) as long as `2·errors + erasures ≤ n − k`.

## What it corrects

| Damage | Handled? |
|--------|----------|
| Erasures (symbols flagged lost) | ✅ up to n − k |
| Symbol errors (wrong value, unknown position) | ✅ up to ⌊(n − k) / 2⌋ |
| Mixed errors + erasures | ✅ while `2·errors + erasures ≤ n − k` |
| Beyond that bound | ✗ `decode()` returns `DT_ERR_DECODE` |

A symbol is corrupted at the *symbol* (8-bit) granularity: any number of bit
errors within one byte is a single symbol error, and a byte flagged lost is a
single erasure.

## API

```c
#include <drifty/rs/rs251.h>

dt_block_encoder *dt_rs_rs251_block_encoder_create(uint16_t n, uint16_t k);
void              dt_rs_rs251_block_encoder_destroy(dt_block_encoder *enc);

dt_block_decoder *dt_rs_rs251_block_decoder_create(uint16_t n, uint16_t k,
                                                   uint16_t s);
void              dt_rs_rs251_block_decoder_destroy(dt_block_decoder *dec);
```

`(n, k)` are fixed at create and require `1 ≤ k ≤ n ≤ 251`. The decoder also takes
a spare-symbols parameter `s` (below). Factories return NULL on a bad argument or
out of memory.

## Buffer layout

Both buffers are `dt_bit`, one symbol per byte, 8 bits MSB-first per byte (see the
block interfaces). They are owned by the codec and have a fixed size:

| Buffer | Contents | Length (bits) |
|--------|----------|---------------|
| **decoded** | the message, as whole bytes | `(k − 1) · 8` |
| **encoded** | the n codeword symbols, one GF(251) value per byte | `8 · n` |

The message holds `k − 1` whole bytes because that is the largest B with
`256^B ≤ 251^k` — one byte less than k symbols, the room GF(251) gives up against
a byte. `decoded_len()` and `encoded_len()` report these in bits.

## Erasures and invalid symbols

On decode, each received 8-bit group of the encoded buffer is mapped to a GF(251)
symbol, **except** that the symbol is treated as an **erasure** when:

- any of its 8 bits is not a clean boolean — i.e. `DT_ERASURE`, `DT_INVALID`, or
  `DT_ABSENT` (e.g. a position an inner decoder judged lost or deleted); or
- the assembled byte is not a valid GF(251) symbol (value `> 250`).

This is the bridge from an inner decoder's output: feed its recovered stream into
the encoded buffer and its lost/absent verdicts become RS erasures automatically.

## Spare check symbols (`s`)

The decoder reserves `s` of the n − k check symbols as a miscorrection guard. A
block spends `2·errors + erasures`; if a decode would leave fewer than `s` check
symbols unspent — `2·errors + erasures > (n − k) − s` — it is rejected as
`DT_ERR_DECODE` even though the algebra could have produced an answer. A larger
`s` trades correction power for a smaller chance of a confident-but-wrong decode;
`s = 0` uses the full n − k budget. Requires `0 ≤ s ≤ n − k`.

## Driving the codec

Drive each side through its block vtable (fill the input buffer, run, read the
output buffer). rs251's `encode()` / `decode()` complete in a single call — they
return `DT_OK` (or a negative error) and never `DT_AGAIN`.

```c
/* encode */
dt_block_encoder *enc = dt_rs_rs251_block_encoder_create(n, k);
dt_bit *msg = enc->decoded_buf(enc);     /* (k - 1) * 8 bits, fill with DT_TRUE/DT_FALSE */
/* ... write the message bits into msg ... */
if (enc->encode(enc) == DT_OK) {
    dt_bit *code = enc->encoded_buf(enc); /* 8 * n bits to transmit */
    /* ... send code ... */
}
dt_rs_rs251_block_encoder_destroy(enc);

/* decode */
dt_block_decoder *dec = dt_rs_rs251_block_decoder_create(n, k, /*s=*/0);
dt_bit *rx = dec->encoded_buf(dec);      /* 8 * n bits; mark lost symbols as erasures */
/* ... write the received symbols into rx ... */
dt_result r = dec->decode(dec);
if (r == DT_OK) {
    dt_bit *out = dec->decoded_buf(dec);  /* (k - 1) * 8 recovered message bits */
    /* ... use out ... */
} else if (r == DT_ERR_DECODE) {
    /* too much corruption (or spare margin not met): block unrecoverable */
}
dt_rs_rs251_block_decoder_destroy(dec);
```

To process another block, overwrite the input buffer, call `reset()`, and repeat.
The full vtable contract (the buffer-ownership and `DT_AGAIN` rules the interface
permits in general) is documented in the [block interface](../block.md).

## Result codes

`encode()` / `decode()` return [`dt_result`](../../include/drifty/result.h):
`DT_OK` on success, `DT_ERR_DECODE` when the block is unrecoverable (corruption
beyond the bound, or the `s` spare margin not met), and `DT_ERR_ARG` on a bad
argument.

## Soft decoding

`dt_rs_rs251_block_soft_decoder_create(n, k, s)` builds a **soft-input** decoder
(the [block soft-decoder interface](../block.md)): its encoded buffer is
`dt_soft_bit` — 8 per codeword symbol — rather than hard `dt_bit`, and it still
produces a hard decoded output. It is driven the same way (fill `encoded_buf()`,
`decode()`, read `decoded_buf()`).

Each symbol's hard value comes from the per-bit argmax, exactly as the hard
decoder reads bits. On top of that, the soft decoder corrects **more than the hard
⌊(n − k) / 2⌋ errors** by an iterative erasure pass: when a decode fails, it erases
the least reliable still-"known" symbol and retries, trading a wrong symbol (which
costs 2 check symbols as an error) for an erasure (1) until `2·errors + erasures ≤
n − k` holds. A symbol's reliability is its **least reliable constituent bit** —
the largest `c_invalid + c_absent` over its 8 soft bits — so a byte an upstream
decoder flagged as poison (`DT_INVALID`) or absent (`DT_ABSENT`) is erased first.
Given good soft flags, this recovers up to `n − k` wrong symbols where the hard
decoder, seeing only the hard values, would fail.

```c
dt_block_soft_decoder *dec = dt_rs_rs251_block_soft_decoder_create(n, k, /*s=*/0);
dt_soft_bit *rx = dec->encoded_buf(dec);   /* 8 * n soft bits */
/* ... write the received soft symbols into rx ... */
if (dec->decode(dec) == DT_OK) {
    dt_bit *out = dec->decoded_buf(dec);   /* (k - 1) * 8 recovered bits */
}
dt_rs_rs251_block_soft_decoder_destroy(dec);
```

## Using it as an outer code

Concatenate it with a `cc/` inner codec: the inner decoder corrects the bulk of
the channel noise and emits output-domain symbols (including `DT_ERASURE` for
values it could not recover and `DT_ABSENT` for positions it judged deleted); rs251
consumes that stream, turns those flags into RS erasures, and cleans up the
residual symbol errors the inner code let through — up to the `2·errors +
erasures ≤ n − k` bound, with the spare margin guarding against miscorrection.
